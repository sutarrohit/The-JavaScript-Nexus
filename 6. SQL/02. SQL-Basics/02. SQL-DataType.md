# **SQL Data Types**

## üß© **Introduction**

In SQL, every column in a database table is designed to hold a specific kind of data.
**SQL data types** define what type of value can be stored in a column.
For example, if a column stores a user‚Äôs age, its type should be `INT` to ensure only whole numbers are allowed.

Choosing the correct data type is critical for:

-   ‚úÖ **Data integrity** ‚Äì preventing invalid data (e.g., no text in numeric fields)
-   ‚öôÔ∏è **Performance** ‚Äì smaller types mean faster queries
-   üíæ **Storage optimization** ‚Äì saves disk space

SQL data types are broadly divided into:

1. **Numeric types** ‚Äì e.g., `INT`, `TINYINT`, `BIGINT`, `FLOAT`, `REAL`
2. **Date and Time types** ‚Äì e.g., `DATE`, `TIME`, `DATETIME`
3. **Character/String types** ‚Äì e.g., `CHAR`, `VARCHAR`, `TEXT`
4. **Unicode String types** ‚Äì e.g., `NCHAR`, `NVARCHAR`, `NTEXT`
5. **Binary types** ‚Äì e.g., `BINARY`, `VARBINARY`, `BLOB`
6. **Miscellaneous types** ‚Äì e.g., `CLOB`, `XML`, `BOOLEAN`, `UUID`, `JSON`

While these are standard, their names, ranges, and behaviors can differ slightly across **MySQL**, **PostgreSQL**, **SQL Server**, and **Oracle**.

---

## üß† **Choosing the Right SQL Data Type**

Selecting the right type affects performance, accuracy, and efficiency.

### üîπ 1. Storage Efficiency

Use the smallest possible type that fits your data range.
For example:

-   Use `TINYINT` (1 byte) instead of `BIGINT` (8 bytes) for small numbers.

### üîπ 2. Performance

Smaller data types can be read faster and processed more efficiently.
Avoid storing numbers as strings.

### üîπ 3. Data Integrity

Data types enforce validation rules.
For instance, a `DATE` column will reject invalid entries like `'Hello'`.

‚úÖ Always refer to your database‚Äôs documentation to confirm supported data types.

## üî¢ **SQL Numeric Data Types**

Used to store numbers (integers or decimals).

| Data Type        | Storage   | Range               | Description                                                                 |
| ---------------- | --------- | ------------------- | --------------------------------------------------------------------------- |
| **TINYINT**      | 1 byte    | 0‚Äì255 (unsigned)    | Very small integers like age or quantity.                                   |
| **SMALLINT**     | 2 bytes   | ‚Äì32,768 to 32,767   | Small integers like years or counts.                                        |
| **INT**          | 4 bytes   | ‚Äì2.1B to 2.1B       | Most common type for whole numbers.                                         |
| **BIGINT**       | 8 bytes   | Huge range          | Used for very large IDs or financial systems.                               |
| **DECIMAL(p,s)** | Variable  | Exact numeric       | For precise financial data. `p` = total digits, `s` = digits after decimal. |
| **NUMERIC(p,s)** | Variable  | Exact numeric       | Same as DECIMAL.                                                            |
| **FLOAT(n)**     | 4/8 bytes | Approximate numeric | For scientific or mathematical calculations.                                |
| **REAL**         | 4 bytes   | Approximate numeric | Single-precision float.                                                     |

üßÆ **Tip:**
Use **DECIMAL/NUMERIC** for currency, **FLOAT/REAL** for scientific data.

## ‚è∞ **SQL Date and Time Data Types**

Used to store dates, times, and timestamps.

| Data Type     | Description                                        |
| ------------- | -------------------------------------------------- |
| **DATE**      | Stores only the date (`YYYY-MM-DD`).               |
| **TIME**      | Stores time only (`HH:MI:SS`).                     |
| **DATETIME**  | Stores both date and time (`YYYY-MM-DD HH:MI:SS`). |
| **TIMESTAMP** | Stores number of seconds since `1970-01-01 UTC`.   |
| **YEAR**      | Stores 2- or 4-digit year (`1901‚Äì2155`).           |

‚úÖ **Use Cases:**

-   `DATE` ‚Üí birthday
-   `DATETIME` / `TIMESTAMP` ‚Üí order creation or login event

## üî§ **SQL Unicode and String Data Types**

Used for storing text and character data.

| Data Type       | Description                  | Unicode | Common Use                         |
| --------------- | ---------------------------- | ------- | ---------------------------------- |
| **CHAR(n)**     | Fixed-length text            | ‚ùå      | State codes, Y/N flags             |
| **VARCHAR(n)**  | Variable-length text         | ‚ùå      | Usernames, titles                  |
| **TEXT**        | Long variable text           | ‚ùå      | Articles, comments                 |

| **NVARCHAR(n)** | Variable-length Unicode text | ‚úÖ      | Names, comments                    |
| **NTEXT**       | Large Unicode text (legacy)  | ‚úÖ      | Long documents (use NVARCHAR(MAX)) |

üí° _MySQL uses_ `VARCHAR` or `TEXT` with `CHARACTER SET utf8mb4` _instead of NVARCHAR._

## üíæ **SQL Binary Data Types**

Used to store **binary data** (not text).

| Data Type          | Description                 | Use Case                   | Max Size    |
| ------------------ | --------------------------- | -------------------------- | ----------- |
| **BINARY(n)**      | Fixed-length binary data    | Fixed hashes or IDs        | 8,000 bytes |
| **VARBINARY(n)**   | Variable-length binary data | Thumbnails, small files    | 8,000 bytes |
| **BLOB**           | Large binary object         | Images, PDFs, audio        | Up to 4 GB  |
| **IMAGE (legacy)** | Older SQL Server type       | Replaced by VARBINARY(MAX) | 2 GB        |

‚ö†Ô∏è For large media, store only the file path or URL ‚Äî not the actual file.

## üíª **SQL Data Types Across Databases**

### üîπ **MySQL**

-   `ENUM`: One value from a predefined list.
-   `SET`: Multiple values from a list.
-   `UNSIGNED`: Prevents negative values (doubles positive range).
-   `YEAR`: 2- or 4-digit year representation.

### üîπ **PostgreSQL**

-   `ARRAY`: Store multiple values in one column.
-   `JSONB`: Binary JSON with indexing.
-   `UUID`: 128-bit unique identifiers.
-   `INET`, `CIDR`: For IP/network storage.

### üîπ **SQL Server**

-   `MONEY`, `SMALLMONEY`: Currency types.
-   `DATETIME2`: High-precision datetime.
-   `UNIQUEIDENTIFIER`: Stores UUIDs.
-   `VARCHAR(MAX)`, `NVARCHAR(MAX)`: Replaces TEXT/NTEXT.

### üîπ **Oracle**

-   `VARCHAR2`: Recommended string type.
-   `NUMBER(p,s)`: General numeric type.
-   `DATE`: Includes both date and time.
-   `CLOB`, `BLOB`, `NCLOB`: Large object storage.

## üîÑ **Type Casting and Conversion**

Converting one data type to another is called **type casting**.

### üî∏ Implicit Conversion

-   Happens automatically by SQL engine.
-   Example: `'123'` ‚Üí converted to integer automatically.

‚ö†Ô∏è Can cause performance issues ‚Äî **avoid when possible.**

### üî∏ Explicit Conversion

-   Manually convert using SQL functions.

#### ‚úÖ `CAST()` ‚Äî ANSI Standard

```sql
CAST(expression AS target_type)
```

#### ‚úÖ `CONVERT()` ‚Äî SQL Server Specific

```sql
CONVERT(target_type, expression, [style_code])
```

## üß± **Best Practices for Choosing Data Types**

### 1. Use the Smallest Type That Fits (Goldilocks Rule)

-   Don‚Äôt default to `INT` if `TINYINT` is enough.
-   Choose `CHAR(5)` for zip codes (to preserve leading zeros).

### 2. Understand Your Data

-   Is it ever negative? Use `UNSIGNED` for positives only.
-   Will it need decimals? Use `DECIMAL` instead of `INT`.

### 3. Plan for Future Growth (Realistically)

-   `INT` is fine for most IDs ‚Äî don‚Äôt overuse `BIGINT`.

### 4. Choose Between CHAR and VARCHAR Wisely

-   `CHAR` for fixed-length (e.g., country codes).
-   `VARCHAR` for variable text (e.g., names, emails).
